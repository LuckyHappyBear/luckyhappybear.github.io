AppWidget开发指南

AppWidget简介

  AppWidget 即桌面小部件，是能直接显示在Android系统桌面上的小程序，是android系统应用开发层面的一部分，但是又有特殊用途，而且会成为整个android系统的亮点。
  我们可以把它认为是app的一个微缩窗口，我们可以通过它来快速的展示一些信息，做一些简单的交互，我们通常将AppWidget嵌入到launcher中，用户不用进入app中即可在桌面进行交互及操作，简单方便，对用户体验是一个较大的提升，目前比较普遍并且常用的小部件比如时钟，天气，音乐播放器等都是十分简单好用的小部件。

AppWidget的简单实现

  在应用中实现一个小部件实际上只要遵循特定的步骤我们很快的就能创建一个简单的小部件，因为系统中的AppWidget框架已经为我们实现了90%的部分，我们只需要按照特定的套路进行实现即可，在实现一个简单的小部件时时我们仅需三步即可完成。

第一步：自定义小部件界面

  首先我们需要定义一个布局文件，你可以按照自己的想法对它进行布局，将他设计成你想要的样子，但是我们在这里需要注意一点，那就是系统默认的小部件布局中支持的布局及组件较为有限，只支持以下布局及组件：

  • 支持的布局
    FrameLayout，LinearLayout，RelativeLayout，GridLayout
  • 支持的组件
    AnalogClock，Button，Chronometer，ImageButton，ImageView，ProgressBar，TextView，ViewFlipper，ListView，GridView，StackView，AdapterViewFlipper

  如上所示，我们定义了小部件的配置信息，接下来我们详细的说一说每个属性的用途:

  • minHeight及minWidth
    这两个属性定义了小部件的最小宽高，当minWidth与minHeight不为桌面cell的整数倍时，小部件的宽高会被调整到最接近的cell的大小，在这里google有一份官方的设计文档，如果是遵循了原生的设计的话，我们的cell及对应的大小应该如下所示：
  • minResizeWidth及minResizeHeight
    这两个属性定义了我们的小部件最小能调整至多大，目前我们的例子中将均其设置为110dp，代表用户在手动设置它的大小时最小只能将其宽高设置为110dp。
  • updatePeriodMillis
    定义了Widget的刷新频率，也就是App Widget Framework多久请求一次 AppWidgetProvider 的 onUpdate() 回调函数。该时间间隔并不保证精确，出于节约用户电量的考虑，Android 系统默认最小更新周期是 30 分钟，也就是说：如果你的程序需要实时更新数据，设置这个更新周期是 2 秒，那么你的程序是不会每隔 2 秒就收到更新通知的，而是要等到30分钟以上才可以，要想实时的更新Widget，一般可以采用Service和 AlarmManager 对Widget进行更新。
  • previewImage
    当用户选择添加 Widget 时的预览图片。如果该属性没有定义，则展示 application 的 launcher icon，该属性是在 3.0 以后引入的。
  • initialLayout
    小部件的布局，也就是我们在第一步中定义的布局。
  • resizeMode
    Widget在水平和垂直方向是否可以调整大小，值可以为：horizontal（水平方向可以调整大小），vertical（垂直方向可以调整大小），none（不可以调整大小），也可以 horizontal|vertical组合表示水平和垂直方向均可以调整大小。
  • widgetCategory
    表示 Widget 可以显示的位置，包括 home_screen（桌面），keyboard（锁屏），keyboard 属性需要 5.0 或以上 Android 版本才可以。

第三步：定义小部件类

  这一部分主要是需要继承系统定义的AppWidgetProvider辅助类，并重写其中的一些方法，我们需要知道的是小部件是通过广播驱动的，AppWidgetProvider本身也是继承自BroadCastReceiver，AppWidgetProvider与Activity一样都使用了模板方法的设计模式，我们要做的就是实现他提供的模板方法。
  AppWidgetProvider 继承自 BroadcastReceiver，内部逻辑非常简单，就是在 onReceive() 中处理 Widget 相关的广播事件，主要有以下广播类型:

  • ACTION_APPWIDGET_UPDATE
  • ACTION_APPWIDGET_DELETED
  • ACTION_APPWIDGET_ENABLED
  • ACTION_APPWIDGET_DISABLED
  • ACTION_APPWIDGET_OPTIONS_CHANGED

  收到这些广播后我们会分别回调以下函数：

  • onUpdate()
  • onDelete()
  • onEnabled()
  • onDisabled()
  • onAppWidgetOptionsChanged()

  我们继承AppWidgetProvider后可以选择性复写除onUpdate()外的所有函数，但是onUpdate()是我们必须要去复写的，因为小部件的更新均在这里实现，想要小部件符合预期，这个函数必须重写定制，接下来我们来说明每个函数的调用时机。

  • onUpdate()
    是最重要的回调函数，widget刷新时就会回调到此函数。
  • onDelete()
    当 Widget 被删除时调用该方法。
  • onEnabled()
    当 Widget 第一次被添加时调用，例如用户添加了两个你的 Widget，那么只有在添加第一个 Widget 时该方法会被调用。所以该方法比较适合执行你所有 Widgets 只需进行一次的操作。
  • onDisabled()
    与 onEnabled 恰好相反，当你的最后一个 Widget 被删除时调用该方法，所以这里用来清理之前在 onEnabled() 中进行的操作。
  • onAppWidgetOptionsChanged()
    当 Widget 第一次被添加或者大小发生变化时调用该方法，可以在此控制 Widget 元素的显示和隐藏。

总结

  经过上面的总结，我们可以发现，实现一个简单的桌面小部件实际上是一个十分轻松的过程，它的实现十分固化，大部分的事情都由系统来管控，我们能做的只是实现相应的回调，但是实际上这些回调的调用时机等我们并不能直接控制，但是在实现一个复杂的小部件时仍有许多内容需要我们注意，数据处理时也要极为小心，这篇总结仅仅是实现了最最基本的内容，我们可以在此基础上进行扩展，开发出好看又使用的桌面小部件应用。
